# 子订单系统接口文档

## 1. 概述

本次改动主要是为遛狗订单（Type=3）增加子订单功能。对于遛狗订单，系统会每天生成一个独立的子订单，每个子订单有自己的状态、开始验证码和结束验证码。

### 订单类型区分：
- **非遛狗订单**（Boarding/Daycare/DropIn）：保持原有流程，使用主订单的验证码
- **遛狗订单**（Walking）：每一天生成一个子订单，使用子订单的独立流程

---

## 2. 订单类型枚举

| type | 服务类型 | 说明 |
|------|----------|------|
| 1 | Boarding（寄养） | 非遛狗订单，使用主订单流程 |
| 2 | Daycare（日托） | 非遛狗订单，使用主订单流程 |
| 3 | Walking（遛狗） | **遛狗订单，使用子订单流程** |
| 4 | DropIn（上门） | 非遛狗订单，使用主订单流程 |

---

## 3. 子订单模型

### SubOrder 对象

```json
{
  "ID": 1,
  "OrderID": 100,
  "Date": "2026-03-14",
  "State": 2,
  "StartCode": "1234",
  "EndCode": "5678",
  "SitterHandleAt": "2026-03-14 10:00:00",
  "SitterFinishAt": "2026-03-14 11:00:00",
  "CreatedAt": "2026-03-14 09:00:00",
  "UpdatedAt": "2026-03-14 09:00:00",
  "DayNumber": 1
}
```

**字段说明：**
- DayNumber: 这是整个订单周期中的第几天，从1开始计数

### 子订单状态（State）

子订单只使用以下4个状态，数字与主订单保持一致：

| 值 | 状态 | 说明 |
|----|------|------|
| 2 | OrderAccepted | 未开始（已接受） |
| 3 | OrderEstablished | 订单开始（输入开始验证码后） |
| 4 | OrderPreCompleted | 预完成（Sitter点击完成） |
| 5 | OrderCompleted | 已完成（输入结束验证码后） |

## 4. 子订单接口

### 4.1 分页查询子订单列表

**接口：** `GET /SubOrder/GetSubOrdersPage`

**Query 参数：**
| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| Current | int | 否 | 当前页，默认1 |
| Size | int | 否 | 每页数量，默认10，最大100 |
| ID | int64 | 否 | 子订单ID |
| OrderID | int64 | 否 | 主订单ID |
| State | int | 否 | 子订单状态 |
| StartTime | string | 否 | 开始时间，格式：2006-01-02 15:04:05 |
| EndTime | string | 否 | 结束时间，格式：2006-01-02 15:04:05 |

### 4.4 Sitter接受/拒绝子订单

**接口：** `POST /SubOrder/SitterHandleSubOrder`

**Content-Type：** `application/json`

**Request Body：**
```json
{
  "OrderID": 1,
  "State": 2
}
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| OrderID | int64 | 是 | 子订单ID（注意：这里是SubOrderID，字段名复用了） |
| State | int | 是 | 状态：2=接受，-1=拒绝 |

**响应示例：**
```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "ID": 1,
    "OrderID": 100,
    "Date": "2026-03-14",
    "State": 2,
    "StartCode": "1234",
    "EndCode": "",
    "SitterHandleAt": "2026-03-14 10:00:00",
    "SitterFinishAt": null,
    "CreatedAt": "2026-03-14 09:00:00",
    "UpdatedAt": "2026-03-14 10:00:00"
  }
}
```

### 4.5 Sitter输入子订单开始验证码

**接口：** `POST /SubOrder/SitterSetSubOrderStartCode`

**Content-Type：** `application/json`

**Request Body：**
```json
{
  "SubOrderID": 1,
  "Code": "1234"
}
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| SubOrderID | int64 | 是 | 子订单ID |
| Code | string | 是 | 开始验证码 |

**响应示例：**
```json
{
  "code": 0,
  "msg": "success",
  "data": null
}
```

**说明：** 验证成功后，子订单状态变为 OrderEstablished(3)，并自动生成结束验证码。

---

**接口：** `POST /SubOrder/SitterSubOrderFinish`

**Content-Type：** `application/json`

**Request Body：**


| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| OrderID | int64 | 是 | 子订单ID（注意：这里是SubOrderID，字段名复用了） |

**响应示例：**
```json
{

  "msg": "success",
  "data": null

```

**说明：** 子订单状态变为 OrderPreCompleted(4)。

---

### 4.7 Sitter输入子订单结束验证码

**接口：** `POST /SubOrder/SitterSetSubOrderEndCode`

**Content-Type：** `application/json`

**Request Body：**
```json
{
  "SubOrderID": 1,
  "Code": "5678"
}
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| SubOrderID | int64 | 是 | 子订单ID |
| Code | string | 是 | 结束验证码 |

**前置条件：** 子订单状态必须为 OrderPreCompleted(4)

**响应示例：**
```json
{
  "code": 0,
  "msg": "success",
  "data": null
}
```

**说明：** 验证成功后，子订单状态变为 OrderCompleted(5)。

## 5. 主订单接口改动说明

### 5.1 创建订单（保持不变）

接口：`POST /Order/CreateOrderWithPricing`

**说明：** 对于遛狗订单（type=3），创建主订单时会自动为 FromDate~ToDate 期间的每一天创建一个子订单，子订单初始状态为 OrderAccepted(2)。

---

### 5.2 Sitter接受/拒绝主订单

接口：`POST /Order/SitterHandleInvite`

**改动说明：**
- 非遛狗订单：保持原有逻辑，接受时生成主订单的开始验证码存入Redis
- 遛狗订单：接受主订单时，每个子订单生成独立的开始验证码存入数据库（StartCode字段），子订单状态保持 OrderAccepted(2)

---

### 5.3 其他主订单验证码接口

以下接口对于遛狗订单会返回错误，提示使用子订单接口：

- `POST /Order/SitterSetCreateCode` → 使用 `POST /SubOrder/SitterSetSubOrderStartCode`
- `POST /Order/SitterFinishOrder` → 使用 `POST /SubOrder/SitterSubOrderFinish`
- `POST /Order/SitterSetFinishCode` → 使用 `POST /SubOrder/SitterSetSubOrderEndCode`

---

## 6. 完整流程示例

### 6.1 遛狗订单完整流程

1. **创建订单**
   - 调用 `POST /Order/CreateOrderWithPricing`，type=3
   - 系统自动创建主订单和每一天的子订单
   - 主订单状态：OrderInitialized(0)
   - 子订单状态：OrderAccepted(2)（已接受/未开始）

2. **Sitter接受订单**
   - 调用 `POST /Order/SitterHandleInvite`，State=2
   - 主订单状态变为 OrderAccepted(2)
   - 每个子订单生成独立的 StartCode（子订单状态保持 OrderAccepted(2)）

3. **每天处理子订单**（以某一天为例）
   - 查询子订单：`GET /SubOrder/GetSubOrdersByOrderId?OrderID={主订单ID}`
   - Sitter输入开始验证码：`POST /SubOrder/SitterSetSubOrderStartCode`
     - 子订单状态变为 OrderEstablished(3)（订单开始）
     - 生成 EndCode
   - Sitter完成子订单：`POST /SubOrder/SitterSubOrderFinish`
     - 子订单状态变为 OrderPreCompleted(4)（预完成）
   - Sitter输入结束验证码：`POST /SubOrder/SitterSetSubOrderEndCode`
     - 子订单状态变为 OrderCompleted(5)（已完成）

### 6.2 非遛狗订单流程（保持不变）

1. 创建订单
2. Sitter接受订单（生成开始验证码）
3. Sitter输入开始验证码
4. Sitter完成订单（生成结束验证码）
5. Sitter输入结束验证码

---

## 7. 数据库表结构

需要创建 `sub_orders` 表：

```sql
CREATE TABLE sub_orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '子订单ID',
    order_id BIGINT NOT NULL COMMENT '主订单ID',
    date DATE NOT NULL COMMENT '日期',
    state INT NOT NULL COMMENT '子订单状态',
    start_code VARCHAR(10) COMMENT '开始验证码',
    end_code VARCHAR(10) COMMENT '结束验证码',
    sitter_handle_at TIMESTAMP NULL COMMENT 'Sitter处理时间',
    sitter_finish_at TIMESTAMP NULL COMMENT 'Sitter完成时间',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    INDEX idx_order_id (order_id),
    INDEX idx_date (date)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='子订单表';
```
